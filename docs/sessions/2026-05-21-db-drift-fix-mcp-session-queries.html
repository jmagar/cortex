User Request

Investigate whether AI sessions are being ingested currently, then trace why an assistant in the axon_rust session concluded they weren't working.

Session Overview

Confirmed AI session ingestion was healthy, traced a false "not working" conclusion in a prior axon_rust session to the recurring Docker named-volume DB drift bug (container reading a different SQLite file than the CLI/ai-watch), fixed it by recreating the container with syslog compose up, and patched the last remaining gap in the dev docker-compose.yml that would have reproduced the drift on any dev build.

Sequence of Events

  1. Ran syslog ai doctor and systemctl --user status syslog-ai-watch.service — confirmed ingestion healthy: 12,004 records, 86 checkpoints, service active 13h, newest indexed entry was the current session
  2. Located the axon_rust session that claimed AI sessions weren't working (d313d646...jsonl) — user message was "can you use the syslog mcp to find the 4 most recent sessions for this project"
  3. Confirmed list_ai_projects, sessions, search_sessions actions exist in dispatch table (src/mcp/tools.rs:45-54) and are wired to real implementations
  4. Queried MCP HTTP endpoint — list_ai_projects returned {"projects":[],"total_projects":0} despite 12,111 rows with ai_project IS NOT NULL in the local DB
  5. Ran syslog compose doctor — identified DB drift: container mounted Docker named volume, CLI/ai-watch used /home/jmagar/.syslog-mcp/data/syslog.db
  6. Searched docs/sessions/ — found this exact problem was hit and fixed twice before (2026-05-16, 2026-05-21 02:40 AM); morning session note explicitly flagged that syslog setup repair doesn't propagate the name: fix to running containers
  7. Verified installed binary and ~/.syslog-mcp/compose/docker-compose.yml both already had the name: fix — running container simply predated it
  8. Ran syslog compose up — container recreated, compose doctor clean, list_ai_projects returns 18 projects
  9. Identified dev docker-compose.yml missing name: on its volume declaration — Compose merge would strip the fix and reproduce drift on any dev build
  10. Added name: line to docker-compose.yml, committed 661a077, pushed

Key Findings

3rd occurrence DB drift has now been hit on 2026-05-16 (v0.25.3 fix), 2026-05-21 02:40 AM (v0.27.2 fix), and 2026-05-21 15:33 (running container predated fixes).
  • src/mcp/tools.rs:45-46,54 — sessions, search_sessions, list_ai_projects all exist and work; the prior assistant made a faulty "not set up" inference from empty results without investigating infrastructure
  • docker-compose.yml:15-17 — top-level volumes: block is only exercised when SYSLOG_MCP_DATA_VOLUME is unset or a bare name; with an absolute path it's a bind mount and the block is ignored entirely
  • src/setup/firstrun.rs:338-353 — write_compose_assets only writes docker-compose.yml from the embedded COMPOSE_ASSET; no separate prod file is ever written to ~/.syslog-mcp/compose/
  • SYSLOG_MCP_VOLUME_NAME controls the Docker volume name only — setting it to a path does not create a bind mount; SYSLOG_MCP_DATA_VOLUME is the bind-mount switch

Technical Decisions

compose up Used syslog compose up over manual docker rm/run — uses same code path as setup repair, ensures env-file substitution is applied correctly.
name: in dev Dev compose extends prod compose but declares its own top-level volumes: block; Compose merges at field level so the dev declaration overwrites the prod name: unless explicitly repeated.
no .env change SYSLOG_MCP_DATA_VOLUME=/home/jmagar/.syslog-mcp/data already set (bind mount active); .env.example already documented both vars — no changes needed.
