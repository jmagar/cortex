## User Request

Expose the syslog MCP tool schema as a resource so agents can discover and consume it via the MCP `resources/list` / `resources/read` protocol without invoking `tools/list`.

## Session Overview

The server now advertises `resources: {}` in its `initialize` capabilities alongside `tools: {}`. Verified with raw JSON-RPC against a fresh binary. ## Key Findings

- `rmcp` 1.6.0 `ReadResourceResult` is `#[non_exhaustive]` — struct literal syntax fails; must use `ReadResourceResult::new(contents)`.
- `Resource = Annotated<RawResource>` has no `From<RawResource>` impl — construct via `Resource::new(raw, None)`.
- `ListResourcesResult` is generated by the `paginated_result!` macro and **does** implement `Default`, so `..Default::default()` works for `next_cursor`.
- The live HTTP server on `:3100` is `/home/jmagar/.claude/plugins/cache/jmagar-lab/syslog/0dcfee27177d/bin/syslog serve mcp` (plugin-cache binary), not the locally built binary. Production won't see the change until that cache is refreshed.
- `docker-compose.yml` had a pre-existing modification (removed `~/.claude-homelab/.env` env_file reference) that was folded into the commit.
- Version sync script validates only `Cargo.toml` — plugin manifests intentionally omit a version field so the GitHub SHA serves as version.

## Technical Decisions

- **URI scheme `syslog://schema/mcp-tool`**: namespaced to the server to avoid collision; descriptive without being overly nested.
- **Return the `tool_definitions()` array directly**: reuses the single source of truth already used by `list_tools`, so schema and resource stay in sync automatically.
- **MIME type `application/json`**: the content is a JSON array of tool definitions; `application/json` is the correct type for agent consumers.
- **Minor version bump (0.11.1 → 0.12.0)**: new MCP capability (`resources`) is user/agent-visible new functionality per CLAUDE.md convention (`feat` → minor).
- **No `plugin.json` version bump**: user clarified that plugin.json should not carry a version field — when installed from GitHub, the commit SHA is the effective version.

## Errors Encountered

**Error 1**: `cannot create non-exhaustive struct using struct expression` on `ReadResourceResult { contents: ..., ..Default::default() }`
- Cause: `ReadResourceResult` is `#[non_exhaustive]`, disallowing struct literal syntax from outside the crate.
- Fix: replaced with `ReadResourceResult::new(vec![...])`.

**Error 2**: `the trait bound ReadResourceResult: Default is not satisfied`
- Cause: same struct literal attempt; `Default` is not implemented.
- Fix: same as above — use the provided constructor.

**Error 3**: `Resource::from(RawResource::new(...))` compile failure
- Cause: `Annotated<T>` has no `From<T>` impl.
- Fix: replaced with `Resource::new(raw, None)` (the two-field struct constructor).

## Risks and Rollback

- **Plugin cache not refreshed**: the live `:3100` server still runs the old plugin-cache binary. Agents connecting to production won't see resources until the cache binary is replaced (`just build-plugin` + process restart).
- **Rollback**: `git revert a8b030e` removes the resource handlers and re-disables the capability with a clean history entry.

## Decisions Not Taken

- **Separate resource per action** (e.g., `syslog://schema/action/search`): rejected — one document per tool is the natural granularity; clients can parse the JSON array themselves.
- **Bumping `plugin.json` version**: explicitly rejected by user — GitHub SHA is the version when installing from a repo source.
